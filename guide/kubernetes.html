# Introduction

Kubernetes deployment is different with traditional microservices deployment. Here, we prefer declarative instead of imperative deployment. Please, read links in references for more information.

# Flow

Example, for staging environment of sent-tv, we have:

  • Application repository: sentadv-users
  • Circle CI: sentadv-users
  • Google Continare Registry: sent-tv-staging
  • Kubernetes Manifest Repo: sent-tv-staging-cluster-01
  • Weave Cloud: sent-tv-staging-cluster-01
  • Kubernetes Cluster: sent-tv-staging-cluster-01

The scenario will be like this:

  1. Developers made changes to application repository. Developers must define their own CI pipeline (Circle CI's config) to push their docker image to GCR.
  2. If the application is new and not listed in Kubernetes Manifest Repository, developers should add Kubernetes manifest needed (deployment and service) by them selves and ask to be merged by others (developers or SRE). Weave Cloud will monitor and apply the changes.
  3. If the application already listed in Kubernetes Manifest Repository, Weave Cloud must be configured to automatically detect new docker image from GCR and apply changes to Kubernetes Cluster and update Kubernetes Manifest Repository.
